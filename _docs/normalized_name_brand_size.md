# `normalized_name_brand_size`

**Generate key ‚Üí ingest ‚Üí record mismatches as variations ‚Üí regenerate translation table ‚Üí next scrape is smarter**

A deterministic/unique string key derived from a product's name, brand, and size. It is the secondary mechanism for recognizing the same product across different supermarkets.

**Model:** `Product.normalized_name_brand_size` ‚Äî `unique`, `db_index`, `max_length=255`.

---

## Why it exists

Barcode is the gold-standard cross-company identifier, but is not always available (Aldi exposes no barcodes). `nnbs` is the fallback. Without it, products from barcode-less companies would never link to their equivalents at other chains, and the optimizer would have far fewer cross-store price comparisons to offer. More matched products = more savings potential for the user.

---

## How it is generated

Generated by `ProductNormalizer.get_normalized_name_brand_size_string()`, called in `BaseDataCleaner._post_process_product()` at scrape time ‚Äî before anything is written to disk.

1. **Extract sizes** from `name`, `brand`, and `size` fields via regex. Standardize units (`grams‚Üíg`, `pack‚Üípk`, `0.095kg‚Üí95g`, etc) and sort.
2. **Normalize brand** ‚Äî strip unicode, lowercase, remove punctuation, sort words (bag-of-words). Look up in `brand_translation_table` and replace with canonical if found.
3. **Combine** `name + normalized_brand + sizes` into one string, apply bag-of-words (`_clean_value`): lowercase, strip punctuation, sort and deduplicate words. This makes the key insensitive to word order and minor formatting differences.
4. **Apply product translation table** ‚Äî if the generated key is a known variation of another product, return the canonical key instead.

---

## How it is used during ingestion

`ProductManager._resolve_products()` uses a three-tier lookup to match incoming products to existing DB rows:

| Tier | Signal | Scope |
|---|---|---|
| 1 | `barcode` | Cross-company |
| 2 | `sku` | Single company |
| 3 | `normalized_name_brand_size` | Cross-company fallback |

Tier 3 is the only tier that can link the same product across different companies without a barcode.

---

## The self-improving loop

This is the key architectural property.

When a product is matched via barcode or SKU but its `nnbs` differs from the canonical product's, `ProductEnricher` records the incoming string in `Product.normalized_name_brand_size_variations`.

After each pipeline run, `ProductTranslationTableGenerator` reads all recorded variations and writes a `variation ‚Üí canonical` mapping to disk. The next scrape fetches this updated table (ETag-cached from the server). `ProductNormalizer` then applies it in step 4 above ‚Äî so future occurrences of that variation resolve to the correct canonical key *before* reaching the DB.

`ProductReconciler` handles late arrivals: any Product rows whose `nnbs` is listed as a variation of another are merged, prices re-pointed, and the duplicate deleted.

```
Scrape ‚Üí translation table applied ‚Üí nnbs generated
  ‚îî‚îÄ Ingestion: Tier 3 lookup catches cross-company matches
       ‚îî‚îÄ Barcode/SKU matches ‚Üí variations recorded
            ‚îî‚îÄ TranslationTableGenerator ‚Üí new mappings written
                 ‚îî‚îÄ ProductReconciler ‚Üí remaining duplicates merged
                      ‚îî‚îÄ [next scrape fetches updated table]
```

Each cycle is slightly more efficient. Every discovered variation becomes a translation that prevents future duplicates.
